# Pattern card — Job Shop
- **Source:** AnyLogic example (models) — distilled, original wording
- **Paradigm:** DES
- **Problem it solves:** Models the end-to-end flow of a small manufacturing facility — from raw material receiving through CNC machining to palletising and outbound truck shipping — so that labour counts and operating costs can be evaluated under different configurations.

## Block chain
Raw material arrives at a receiving dock via periodic truck deliveries (injected in batches of `TruckCapacity`). Parts move into receiving storage, then forklifts transport them to one of three parallel CNC machines (A, B, C) for machining. After processing, finished parts are collected onto pallets. Full pallets are staged in a shipping-side storage area, where a truck periodically collects them. Two separate Sink blocks terminate the receiving truck flow and the outbound shipping truck flow. A `TimeMeasureStart / TimeMeasureEnd` pair brackets the pallet journey to record cycle time. A `Hold` block gates pallet release, letting exactly `TruckCapacity` units accumulate before the outbound truck departs. `RestrictedAreaStart / RestrictedAreaEnd` blocks enclose forklift movement zones to enforce path constraints in the 3-D layout.

## Resources
- **ForkLift resource pool** — quantity controlled by the `NForkLifts` slider (range 1–5, default 2); forklifts navigate the 3-D space at `uniform(ForkLiftSpeed * 0.8, ForkLiftSpeed * 1.2)` to model speed variation.
- **CNCMachine agents** — three parallel workstation instances (A, B, C), each represented as a sub-agent class.
- **Workers** — visible in the 3-D scene (sitting and office worker 3-D models) but tracked indirectly through labour utilisation charts rather than as an explicit resource pool.
- **Trucks** — inbound (raw material) and outbound (finished pallets) truck agents generated by Source blocks on independent schedules.

## Key settings worth copying
- **Processing time:** `triangular(ProcessingTimeMin, ProcessingTimeMean, ProcessingTimeMax)` — same distribution applied identically to all three CNC machines; min/mean/max exposed as user-editable parameters.
- **Batch gating:** `Hold` block releases flow only when `in.count() % TruckCapacity == 0`, ensuring the outbound truck always departs full.
- **Forklift speed jitter:** `uniform(ForkLiftSpeed * 0.8, ForkLiftSpeed * 1.2)` — ±20 % variation around the slider value; easy to replicate for any mobile resource.
- **Batch injection:** `sourcePallets.inject(TruckCapacity)` called on the inbound truck arrival event rather than using a timed rate, linking physical truck arrivals to material flow.
- **Model time unit:** Seconds — fine-grained enough for short machining cycles and forklift travel.
- **Slider ranges:** NForkLifts 1–5; TruckSpeed 0–100 m/s; ForkLiftSpeed 0–100 m/s; PalletCapacity 0–100; all wired to parameters so experiments need no code changes.

## KPIs instrumented
- **Storage utilisation by minutes** — tracked for both receiving and shipping storage areas.
- **Forklift utilisation** — `ForkLift.utilization()` displayed on the metrics dashboard.
- **CNC machine utilisation** — per-machine utilisation percentage charted over time.
- **Throughput** — `throughput` statistic collected at each CNC machine workstation block and at the shipping Sink.
- **Operating cost** — charted as a function of the chosen labour/equipment configuration (labour quantities and speeds).
- **Cycle time** — measured via `TimeMeasureStart / TimeMeasureEnd` around the pallet journey.

## Reusable idea
Use a `Hold` block with a modulo counter (`count() % batchSize == 0`) to simulate batch-release logistics — trucks, bins, containers, or kanbans — without needing a separate Batch block or custom agent logic. Pair it with an `inject()` call on the arrival event of the carrier agent to keep the material flow tightly coupled to the physical vehicle schedule.
