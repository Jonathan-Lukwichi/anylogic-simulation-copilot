# Pattern card — Maintenance of a Coffee Machine
- **Source:** AnyLogic example (basicmodels) — distilled, original wording
- **Paradigm:** DES
- **Problem it solves:** Modelling multiple coexisting downtime types (periodic maintenance, unplanned repair, and on-demand custom tasks) on a single piece of equipment, and linking those downtimes to resource task processes.

## Block chain
The model centres on a single equipment agent (CoffeeMachine) that runs a coffee-serving flow — Source → Seize (assistant resource) → Delay (service) → Release → Sink — while several independent Downtime blocks interrupt that flow in parallel:

1. **Periodic filter change** — fires on a recurring schedule (`uniform_discr(20, 25)` minutes between cycles); the actual swap takes `triangular(1, 2, 2.5)` minutes and seizes the assistant resource, so maintenance and customer service compete for the same person.
2. **Water-tank refill** — another scheduled Downtime (`uniform_discr(13, 15)` minute period, `uniform_discr(4, 5)` minute duration) that does not require the assistant, showing resource-independent downtime.
3. **Coffee-grounds removal** — a third periodic Downtime (`uniform_discr(13, 15)` minute period, `triangular(1.5, 2, 3)` minute task) that feeds into a sub-process: the assistant is seized, grounds are removed (`triangular(3, 5, 8)` min) then a new filter is installed (`triangular(5, 6, 8)` min), demonstrating a chained multi-step maintenance sequence inside one downtime event.
4. **Machine relocation** — triggered manually via a UI button rather than a schedule; a custom Downtime block kicks off a ResourceTaskStart → MoveTo → ResourceTaskEnd chain (`uniform_discr(10, 20)` min travel, `uniform_discr(15, 30)` min setup), illustrating how a Downtime can wrap a full spatial resource task.
5. **Unplanned breakdown** — a failure-and-repair Downtime with `uniform_discr(20, 30)` minute MTBF and `triangular(2, 2.5, 3)` minute repair, representing random equipment failure.

Customer requests are generated by a Source block, served through a Service block (`triangular(1, 2, 3.5)` minutes), and pass through RestrictedArea blocks that prevent new customers from entering while the machine is in a downtime state. A Queue block buffers arrivals that arrive while the machine is unavailable.

Model time unit: **Minute**.

## Resources
- **Assistant** — 1-unit ResourcePool; seized for filter changes and grounds removal; competes with customer service demand.
- **CoffeeMachine agent** — the equipment entity that owns all Downtime blocks; also serves as a movable resource unit in the relocation scenario.

## Key settings worth copying
| Setting | Value |
|---|---|
| Filter-change interval | `uniform_discr(20, 25)` min |
| Filter-change duration | `triangular(1, 2, 2.5)` min |
| Water-tank refill interval | `uniform_discr(13, 15)` min |
| Water-tank refill duration | `uniform_discr(4, 5)` min |
| Grounds-removal interval | `uniform_discr(13, 15)` min |
| Grounds + filter task | `triangular(3,5,8)` + `triangular(5,6,8)` min |
| Breakdown MTBF | `uniform_discr(20, 30)` min |
| Repair time | `triangular(2, 2.5, 3)` min |
| Relocation travel | `uniform_discr(10, 20)` min |
| Relocation setup | `uniform_discr(15, 30)` min |
| Customer service time | `triangular(1, 2, 3.5)` min |
| Time unit | Minute |

RestrictedArea blocks gate customer entry during downtime — a critical pairing with any Downtime block to avoid entities entering an unavailable resource.

## KPIs instrumented
- Machine availability / downtime fraction (implicit via Downtime block statistics)
- Queue length and customer wait time during unavailability windows
- Assistant utilisation (shared across service and maintenance tasks)
- Throughput (customers served per hour via Sink statistics)

## Reusable idea
**Layer multiple named Downtime blocks onto one resource to represent distinct failure/maintenance modes independently.** Each Downtime has its own schedule, duration distribution, and optional resource seizure — they interrupt the main process flow without needing a single combined failure model. Pairing a Downtime with ResourceTaskStart/End lets a maintenance event trigger a full spatial or multi-step task chain, and wrapping the main flow entry in RestrictedArea blocks cleanly prevents new work from entering while any downtime is active.
