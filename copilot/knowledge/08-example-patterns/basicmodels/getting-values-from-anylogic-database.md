# Pattern card — Getting Values From AnyLogic Database
- **Source:** AnyLogic example (basicmodels) — distilled, original wording
- **Paradigm:** DES
- **Problem it solves:** Drive simulation parameters from a built-in database table so that scenario values and per-entity processing times can be changed without touching model logic.

## Block chain
Trucks (agent type) are generated by a **Source** block whose inter-arrival time comes from a model parameter. Agents flow into a **Queue**, then into a **Delay** block that represents the unloading operation. On exit they reach a **Sink**. The Delay's service duration is not hard-coded: it is resolved at runtime by querying the `truck_processing_times` database table, selecting the matching row by the truck's `cargoType` attribute. A separate `scenarios` table stores named parameter presets; the active scenario name is selected from a combo-box control, and the query `SELECT_UNIQUE_VALUE` pulls the corresponding parameter value into the model.

## Resources
- Single **Delay** block acting as the unloading station (capacity parameter read from the database)
- No explicit ResourcePool; capacity is a scalar integer loaded from `truck_processing_times`

## Key settings worth copying
- `SELECT_UNIQUE_VALUE` database query type on the Delay's delay-time field — returns one cell from a filtered row
- `selectFrom(truck_processing_times).orderBy(truck_processing_times.unloading_time.asc()).firstResult(truck_processing_times.cargo_type)` — fluent query API for sorted lookup
- Scenario switching via a combo-box whose list is populated with `SELECT name FROM scenarios;` — lets the user swap full parameter sets at runtime without recompiling
- Model time unit: **Hour**
- Agent population initialised with `ONE_AGENT_PER_DATABASE_RECORD` (not used for the main flow here, but available pattern for pre-loading agent attributes from a table)

## KPIs instrumented
- Throughput (trucks processed through Sink)
- Queue content / wait time (implicitly visible on Queue block statistics)
- Delay utilisation (Delay block built-in stats)

## Reusable idea
Store scenario configurations and entity-level parameters in AnyLogic's built-in database tables, then use `SELECT_UNIQUE_VALUE` queries directly inside block fields. This cleanly separates data from model structure: analysts can add new cargo types or scenarios by editing table rows in the Database view rather than modifying Java code or block properties, making sensitivity analysis and what-if comparisons trivial to set up.
