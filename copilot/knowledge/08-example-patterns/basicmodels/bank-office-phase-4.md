# Pattern card — Bank Office - Phase 4
- **Source:** AnyLogic example (basicmodels) — distilled, original wording
- **Paradigm:** DES
- **Problem it solves:** Models a retail bank branch where arriving customers either join a teller queue or use an ATM, measuring end-to-end wait and service times under a time-varying arrival schedule.

## Block chain
Customers are generated by a **Source** whose arrival rate follows a schedule (`rateSchedule`) rather than a fixed value. On entry, a **TimeMeasureStart** stamps each customer. A **SelectOutput** routes them probabilistically — some head to an **ATM Delay** block (triangular service time), others join a **Queue** feeding into **Service** blocks staffed by tellers. A second **SelectOutput** inside the teller path handles any priority or routing logic between teller stations. **TimeMeasureEnd** blocks placed at each exit path capture sojourn time before customers leave through a **Sink**.

## Resources
- **ResourcePool** of Teller agents (capacity driven by a schedule or fixed integer; `capacitySchedule` / `capacityOnValue` parameters present) — tellers are seized inside **Service** blocks and released on completion.
- ATM modeled as a standalone **Delay** (no resource pool; capacity parameter governs simultaneous users).

## Key settings worth copying
- Arrival distribution: schedule-driven rate (`rateSchedule`) — swap to `exponential(rate)` for simpler Poisson arrivals.
- ATM service time: `triangular(0.8, 1.5, 3.5)` minutes — min/mode/max spread reflects quick self-service.
- Teller service time: `triangular(2.5, 6, 11)` minutes — wider spread captures complex transactions.
- Time unit: **Minute** (set at model level).
- Queue discipline: priority parameter exposed on Queue block — easy to switch FIFO to priority-based.
- Agent type: custom **Customer** agent class; **Teller** agent class for resource pool members.

## KPIs instrumented
- **Sojourn / wait time** — captured by paired TimeMeasureStart / TimeMeasureEnd blocks on both the teller path and the ATM path independently.
- **Queue length** — monitored via Queue block's built-in statistics.
- **Resource utilisation** — ResourcePool tracks teller busy fraction automatically.

## Reusable idea
Use a **time-varying arrival schedule** on the Source rather than a constant rate, then split the flow immediately with SelectOutput so two service channels (self-service ATM vs. staffed teller) run in parallel with separate KPI probes — this pattern cleanly separates measurement concerns for each channel without duplicating the whole process chain.
